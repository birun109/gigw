GIGW 3.0 Statement 5.3.1: Security Audit and Clearance

Analysis and Requirements

**GIGW Requirement****Security Audit and Clearance Certificate**
**Audit Authority**NIC / STQC / CERT-In empanelled laboratories
**Audit Baseline**OWASP Top 10 vulnerabilities (latest version), OWASP ASVS, OWASP MAVS
**Goal**Ensure confidentiality, integrity, and availability (CIA) of government services before public deployment.
**Mandate****Mandatory** clearance certificate before hosting in the production environment.

Developer Action: Securing Code and Configuration

Key Code & Configuration Security Checklist

  • **Software Updates:** Ensure all CMS, third-party plugins, and underlying code are updated to the **latest secure versions**.
  • **Credential Security:** All passwords, tokens, and keys must be stored using **salted hash encryption**, never in plain text.
  • **Error Handling:** Display **custom error pages** for exceptions; never expose source code, database errors, or server details.
  • **Input Validation:** **Sanitise user input** at both the client and server end (syntactical and semantic).
  • **Session/Cookie Management:** Cookies must be **Secure** and **HTTP Only**.
  • **Account Privilege:** Web Server processes must run under a dedicated user with **limited privileges**, not Administrator/Root.
  • **Access Control:** Implement **Role-Based Access Control (RBAC)** and the principle of **minimal privilege** for all system users.

Developer Action: Securing Databases

Key Database Security Checklist

  • **Encryption:** Implement **strong encryption** and key management for data **at rest** and **in transit** (using TLS/SSL).
  • **Credentials:** Use strong, unique credentials for application access. **Remove/change all default administrative passwords**.
  • **Network Access:** Database should only be accessible from the **Web Server** on a **whitelisted port**. **Never assign a publicly accessible IP**.
  • **Privilege:** The application must use the **lowest possible level of privilege** when accessing the database.
  • **Surface Reduction:** Turn off all unnecessary database functionality, services, and utility packages.
  • **Auditing & Logging:** Enable and periodically monitor audit trail logs and database logs for suspicious activity.

Example for Compliance Criteria ✅ (Audit Clearance)

Success Scenario: Deployment Clearance

Action: The development team completes the web application and runs automated security scans. They submit the application to a CERT-In empanelled lab for a full VAPT (Vulnerability Assessment and Penetration Testing).

The lab issues a report confirming that all critical and high-severity findings related to the OWASP Top 10 have been mitigated, and a **"Security Audit Clearance Certificate"** is issued. The application is then moved to the production environment.

**Compliance Justification:** Compliance is achieved by the **mandatory successful security audit by an authorized body** before public hosting, ensuring the necessary security standards are met.

Example for Failure Criteria ❌ (No Audit)

Failure Scenario: Deployment Without Clearance

Action: The government organization pressures the development team to launch the portal quickly due to a deadline. The application is deployed to the production environment based only on internal testing and a self-certified checklist.

**Compliance Justification:** Deploying the website, application, or portal without an official **Audit Clearance certificate** from a designated authority (NIC/STQC/CERT-In empanelled lab) is a direct violation of GIGW Statement 5.3.1, regardless of how robust the internal security practices are.

Summary of Conformity

GIGW Requirement Key Mandate
5.3.1 Security Audit and Clearance **Obtain a formal Security Audit Clearance Certificate** from an approved agency (NIC/STQC/CERT-In empanelled lab) **BEFORE** hosting any new or substantially modified website/application in the public production environment.
