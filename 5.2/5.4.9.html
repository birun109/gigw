GIGW 3.0 Statement 5.4.9: Accessible Document Formats

Analysis and Requirements

**Requirement Focus**Provision of documents in accessible formats.
**Mandatory Formats**HTML or **other accessible formats** (e.g., properly tagged PDF).
**Government Action**Ensure the accessible version of **all scanned documents** is available.
**Developer Action**Ensure all uploaded documents are in accessible formats.
**Key Benefit**Enables visually challenged users to use assistive technologies (Screen Readers) to read and navigate content.

Examples of Accessible Documents

  • **HTML:** Best option for maximum accessibility and responsiveness.
  • **Tagged PDF (Mandatory):** PDFs must be created with a proper structure (tags, headings, alt text) recognized by screen readers.
  • **Accessible DOCX/ODT:** Documents created in word processors with correct heading structures, tables, and alt text.

Example for Success Criteria ✅

Success Scenario 1: Providing Accessible Versions

Action: A new budget circular is prepared. The source document is a PDF.

  • The Developer uploads the **Tagged PDF** which was checked for accessibility tags (alt text for images, proper heading hierarchy).
  • **OR** A dedicated **HTML page** containing the exact text of the circular is created and linked directly below the PDF link.
  • For an old **scanned document**, the government organization ensures the text is transcribed into a new, accessible Word document or HTML page and linked alongside the scanned image.

**Compliance Justification:** The content is available in a format that assistive technologies can interpret (Tagged PDF or HTML), and the requirement to ensure scanned documents have an accessible version is met. This is fully compliant with GIGW 5.4.9.

Example for Failure Criteria ❌

Failure Scenario 1: Reliance on Scanned PDFs

Action: The "Archive of Policy Orders" section contains only documents that were scanned directly from old printed copies and saved as PDFs. They have no embedded text layer or accessibility tags.

**Compliance Justification:** The scanned PDFs are essentially **images of text**, which screen readers cannot interpret. This violates the government action to ensure the **accessible version of all scanned documents** is available, failing GIGW 5.4.9.

Failure Scenario 2: Untagged Documents

Action: A new document is uploaded as a PDF. While not scanned, the PDF was exported from a design program without adding any logical structure or accessibility tags (e.g., the title is not marked as a H1, and tables are unstructured).

**Compliance Justification:** Although it is a digital file, an **untagged PDF** is not considered an "accessible format" because assistive technologies cannot logically parse the content structure. This omission violates the developer action to ensure uploading in accessible formats, failing GIGW 5.4.9.

Summary of Conformity

GIGW Requirement Key Mandate
5.4.9 Accessible Document Formats Documents must be provided in **HTML** or a format that assistive technologies can read (e.g., **properly Tagged PDF**). For **scanned documents**, a separate, accessible text version must always be provided to ensure compliance.
